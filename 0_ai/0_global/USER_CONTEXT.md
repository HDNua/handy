## 2. 핵심 프로젝트 자산: DataMover(DM) 검증 프레임워크

- **프로젝트명**: DMA(DataMover) 프로토콜 및 데이터 무결성 검증 자동화

### 하드웨어 도메인 지식

- **DataMover**: 모뎀 IP(클라이언트)와 DRAM 사이 직접 메모리 접근을 담당하는 전용 DMA 컨트롤러. CPU 부하 경감이 목적.
- **프로토콜 복잡도**:
  - **Flow Control**: Ready/Valid 핸드셰이크 방식. 안전하지만 클라이언트를 throttle할 수 있음
  - **Non-Flow Control**: 백프레셔 없는 스트리밍 방식. 빠르지만 덮어쓰기 위험 존재. 최소 전송 간격 보장 필수
- **도메인 제약**:
  - 전송 간격 최대: **1ms (1 TTI)**
  - Non-Flow Control 클라이언트는 최소 간격 보장 필요 (예: 20us)

### 검증 전략 ("4가지 재료" 접근법)

1. **클라이언트 데이터 (`*_DM_test.txt`)**: Checker IP가 생성한 골든 레퍼런스
2. **시뮬레이션 로그**: DM 인터럽트 이벤트가 포함된 모뎀 TOP 시뮬레이션 로그
3. **AXI 트랜잭션 로그 (`dram_cp2ap.txt`)**: 사이클 정확도의 AXI 신호 덤프
   - 문제: 데이터가 byte-enable 신호(`STRB`)와 함께 단편화된 burst로 수신됨
   - 해결(`s03_get_dram_data`): Python으로 **바이트 병합 및 논리 재구성** 알고리즘 구현 → 단편화된 AXI burst를 64바이트 블록으로 복원
4. **DM 스펙**: 설정 맵 (Base Addr, Flow Control 모드, 인터럽트 매핑)

### 문제 상황

- 기존 **Perl** 스크립트는 "쓰기 전용 코드" (유지보수 불가)
- 검증이 시뮬레이션 파형을 눈으로 확인하는 방식에 의존 → 복잡한 타이밍 위반 미검출
- 특정 문제: Flow Control 클라이언트가 인터럽트를 과도하게 발생시키면 기존 스크립트가 동기화를 잃음 (Error 2.2)

### 해결책 (아키텍처)

- **Python 전환**: **OOP** (Parser-Runner) 기반으로 재설계. 모듈 `s00`~`s15`로 구성
- **자동 에러 시그니처 분석**:
  - `INTERVAL_ERROR` (s11): Non-Flow 클라이언트 최소 간격 위반 (FIFO 오버플로우 가능성)
  - `DATA_UNMATCHED` (s12): 핵심 데이터 무결성 검사
  - **휴리스틱 타이밍 검사 (s15)**: 시뮬레이션 시간과 실제 시간 드리프트로 인한 false positive 필터링을 위한 적응형 간격 검사 (예: 1200ns vs 1500ns)
  - `NO_DATA_IN_MEM`: 데이터 유실 감지 (클라이언트 전송했으나 DRAM 미수신)
  - `NO_INTERRUPT` / `NO_BURST`: 프로토콜 핸드셰이크 실패 (페어링 누락)
- **엔지니어링 효율**:
  - **DX**: **VSCode + TDD** 환경 도입 → 디버그 시간 **50%+** 단축
  - **최적화**: GB급 로그 파일에 `fseek` tail-reading 구현

### 성과

- "블랙박스 레거시 스크립트"를 **"투명하고 유지보수 가능한 검증 프레임워크"** 로 전환
- 이후 프로젝트(Project 7 등)에서 DMA 데이터 손상 관련 **Zero-Defect** 달성
