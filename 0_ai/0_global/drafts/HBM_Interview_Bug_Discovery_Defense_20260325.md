# HBM Interview Bug Discovery Defense

## 목적
SK하이닉스 `HBM Digital Design(검증)` 경력 면접에서,  
`Bitmap Waveform Viewer를 통한 Latch bug 탐지`와  
`Register Functional Coverage 기반 RTL bug 탐지` 성과를  
과장 없이 안전하게 설명하기 위한 방어 논리를 정리한다.

## 배경
두 사례 모두 실제 성과는 맞다.

1. Bitmap Waveform Viewer 개발 이후 기존 방식으로 놓치던 이상 징후를 시각화했고, 이후 Latch timing bug로 수렴되었다.
2. Register Functional Coverage 개발 이후 기존 검증에서 놓치던 조합 이슈를 드러냈고, RTL bug 1건 탐지로 이어졌다.

다만 현재 상태에서 주의할 점은 다음과 같다.

- bug의 최종 RTL root cause를 본인이 완전히 설명할 수 있는 상태는 아닐 수 있다.
- 그런데 서류에서 `bug 탐지`라는 표현은 면접관이 높은 확률로 깊게 파고들 수 있는 포인트다.
- 따라서 면접에서는 `무엇을 발견했고 어떤 방식으로 bug를 가시화했는가` 중심으로 답변해야 한다.

## 핵심 원칙
### 1. 본인의 기여를 정확히 정의한다
- `내가 RTL bug를 최종 수정했다`가 아니다.
- `내가 기존에 보이지 않던 이상 징후를 드러냈고, bug로 확인될 수 있게 검증 관점에서 기여했다`가 맞다.

### 2. root cause를 모르면 모른다고 전제하되, 역할은 분명히 말한다
- 설계팀이 최종 원인을 규명하고 수정했을 수 있다.
- 본인은 이상 징후 발견, 문제 구간 축소, 재현성 확보, 설계팀 전달, 수정 후 재검증에 기여한 것으로 설명한다.

### 3. 검증 엔지니어의 가치로 설명한다
- 검증 직무에서 중요한 것은 `최종 회로 설계 수정` 자체보다,
  `이상 징후를 체계적으로 드러내고 재현 가능하게 만들며 sign-off 리스크를 줄이는 것`이다.

## 면접관이 왜 이 부분을 물을 수 있는가
다음 조건을 만족하는 성과는 기술면접에서 자주 타깃이 된다.

1. 숫자와 결과가 분명하다
2. 도구 개발과 bug 탐지가 연결되어 있다
3. 일반적인 업무 나열보다 진위 확인이 쉽다
4. 지원자의 깊이와 과장 여부를 가리기 좋다

즉, 이 두 사례는 면접관 입장에서 `한 번 찔러보기 좋은 대표 성과`다.

## 안전한 포지셔닝 문장
### 기본 프레임
`제가 강조하고 싶은 부분은 RTL root cause를 제가 단독으로 모두 규명했다는 점이 아니라, 기존 방식으로는 놓치던 이상 징후를 제 도구와 검증 관점으로 가시화해서 bug로 수렴되도록 만든 점입니다.`

### Bitmap Waveform Viewer용 프레임
`텍스트 로그만으로는 놓치기 쉬운 timing mismatch 구간을 시각화해서 문제 구간을 빠르게 좁혔고, 그 결과 설계 측에서 latch timing issue로 확인 및 수정할 수 있었습니다. 저는 수정 전후 동작 차이를 다시 검증하는 역할까지 수행했습니다.`

### Functional Coverage용 프레임
`제가 한 핵심 기여는 spec 기반 조합을 coverage 관점으로 모델링하고, 기존 검증에서 보이지 않던 미검증 또는 이상 조합을 드러내는 도구를 만든 것입니다. 그 결과 설계 측과 공유 가능한 형태로 issue를 수면 위로 올렸고, RTL bug로 확인된 사례가 있었습니다.`

## 위험한 답변 패턴
다음 표현은 피한다.

1. `제가 latch bug를 잡았습니다.`
2. `제가 RTL bug를 찾고 수정했습니다.`
3. `정확한 원인은 모르는데 아무튼 bug였습니다.`
4. `제가 다 했습니다.`
5. `HBM 가서도 이런 식으로 바로 bug 잡을 수 있습니다.`

위 표현들은 꼬리 질문에 매우 취약하다.

## 면접관의 예상 꼬리 질문
### Bitmap Waveform Viewer 관련
1. 기존 로그 방식으로는 왜 못 잡았나요?
2. Viewer에서 정확히 무엇이 달라 보였나요?
3. mismatch는 어떤 symptom으로 나타났나요?
4. 그걸 왜 bug라고 판단했나요?
5. 수정 후 무엇으로 재검증했나요?
6. 본인 역할과 설계팀 역할은 어떻게 나뉘었나요?

### Functional Coverage 관련
1. 어떤 coverage 항목 또는 조합에서 이상이 발견됐나요?
2. 기존 regression에서는 왜 놓쳤나요?
3. coverage hole과 RTL bug를 어떻게 구분했나요?
4. issue를 어떤 근거로 설계팀에 전달했나요?
5. 수정 후 coverage와 동작이 어떻게 달라졌나요?
6. 본인이 한 일은 도구 개발인지, bug 분석인지, 둘 다인지?

## 답변 시 강조해야 할 포인트
### Bitmap Waveform Viewer
1. 텍스트 로그 기반 디버깅의 한계가 있었다
2. Viewer가 타이밍/순서 이상을 직관적으로 보여줬다
3. 문제 구간을 좁히는 데 결정적 역할을 했다
4. 설계 측과 협업하여 issue가 latch timing bug로 확인되었다
5. 본인은 검증 관점의 가시화 및 재검증에 강점이 있다

### Functional Coverage
1. 기존 simulation 기반 검증에서 보이지 않는 미검증 조합이 있었다
2. spec 기반 조합을 coverage 대상으로 재구성했다
3. 도구가 누락 또는 이상 조합을 먼저 드러냈다
4. 그 정보를 설계 측과 공유 가능한 근거로 정리했다
5. 본인은 coverage를 숫자 관리가 아니라 검증 공백 탐지 수단으로 본다

## 지금 당장 써먹을 수 있는 1차 방어 답변
### Q. "Viewer로 latch bug를 잡았다고 하셨는데 정확히 어떤 bug였나요?"
`정확히 말씀드리면 제가 RTL root cause를 단독으로 규명했다기보다, 기존 텍스트 로그만으로는 놓치던 timing mismatch 구간을 Bitmap Viewer로 시각화해서 문제 구간을 빠르게 좁힌 사례입니다. 그 결과 설계 측에서 latch timing issue로 확인하고 수정할 수 있었고, 저는 수정 전후 동작 차이를 다시 검증하는 역할을 했습니다. 제가 강조드리고 싶은 성과는 bug의 회로 원인을 과장하는 것이 아니라, 기존에 안 보이던 문제를 보이게 만든 검증 도구를 만들었다는 점입니다.`

### Q. "Functional Coverage로 RTL bug를 찾았다고 했는데 어떤 의미인가요?"
`제가 한 핵심 작업은 spec 기반 레지스터 조합을 coverage 관점에서 모델링하고, 실제 vector에서 어떤 조합이 빠지거나 비정상적으로 동작하는지를 드러내는 도구를 만든 것입니다. 그 과정에서 기존 검증에서는 잘 드러나지 않던 이슈를 발견했고, 설계 측과 공유한 뒤 RTL issue로 confirm된 사례가 있었습니다. 즉 제가 강조하는 포인트는 제가 모든 root cause를 단독으로 분석했다기보다, coverage를 통해 검증 사각지대를 실제 issue로 연결했다는 점입니다.`

## 복원 필요 항목 체크리스트
면접 전 아래 6가지는 반드시 각 사례별로 복원해둘 것.

1. 최초 symptom은 무엇이었는가
2. 기존 방식으로는 왜 놓쳤는가
3. 본인 도구/coverage가 어떻게 문제를 드러냈는가
4. 본인이 실제로 한 액션은 무엇이었는가
5. 설계팀 또는 타팀이 한 액션은 무엇이었는가
6. 최종적으로 어떻게 bug로 닫혔는가

## 현재 시점의 현실적인 전략
이 두 사례는 버릴 필요는 없다. 오히려 강한 성과다.  
다만 `내가 bug를 다 이해했고 끝까지 다 했다`가 아니라,

- `내가 이상 징후를 구조적으로 드러냈다`
- `문제 구간을 좁히고 설계팀과 연결했다`
- `수정 전후 검증까지 기여했다`

이 프레임으로 가져가는 것이 가장 안전하다.

## 후속 작업 제안
다음 단계에서는 아래 두 항목을 별도로 복원하면 좋다.

1. `Bitmap Waveform Viewer -> Latch bug` 사례 1분 답변
2. `Functional Coverage -> RTL bug` 사례 1분 답변
