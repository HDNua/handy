1. 결론부터 짧게 말하고, 사례를 붙이고, 마지막에 HBM 직무 연결성으로 마무리한다.
2. HBM, D2D, PCIe, CXL, Co-emulation 직접 경험은 없는 만큼 절대 있는 것처럼 말하지 않는다.

## 실전 추궁 포인트 10개

### 1. "HBM 직접 경험이 없는데 왜 HBM 검증에 적합하다고 보셨나요?"
**질문 의도**
- 가장 큰 공백 검증
- 도메인 전환 리스크 확인

**답변 초안**
직접적인 HBM 프로젝트 경험은 없습니다. 대신 8년 동안 Modem Block/TOP 검증을 하면서 DataMover(DMA), Memory Power Gating, Register Coverage, VP 교차 검증처럼 복잡한 디지털 검증 문제를 지속적으로 다뤄왔습니다. 제가 강점이 있는 부분은 특정 제품명이 아니라, Spec을 읽고 검증 항목으로 분해한 뒤 Testbench, Checker, Regression, Coverage까지 연결하는 검증 방법론 자체입니다. 그래서 HBM 고유 구조는 빠르게 학습하되, 검증 프레임 자체는 바로 기여할 수 있다고 판단해 지원했습니다.

**주의 포인트**
- `HBM도 사실 비슷합니다`처럼 가볍게 말하지 말 것
- `직접 경험은 없지만, 검증 문제를 구조화하는 방식은 바로 적용 가능하다`로 정리

### 2. "DataMover(DMA) 검증 경험이 HBM이나 Memory Controller 검증과 어떻게 연결되나요?"
**질문 의도**
- 지원동기에 쓴 `메모리 인터페이스 경험`의 진짜 깊이 확인
- DMA 경험이 단순 로그 분석 수준인지, 인터페이스 검증 사고까지 있는지 확인

**답변 초안**
DataMover 검증에서 가장 중요했던 것은 데이터가 메모리로 언제, 어떤 순서로, 어떤 제약 아래 전달되는지를 검증하는 일이었습니다. 저는 Flow/Non-Flow Control 시나리오를 나눠서 인터럽트, 전송 간격, 주소 정보, AXI 로그를 함께 보고 데이터 정합성을 확인하는 Checker와 Scoreboard 성격의 로직을 운영했습니다. HBM 자체 프로토콜을 직접 다뤄보진 않았지만, 메모리 경로에서 발생하는 타이밍 제약, 데이터 무결성, 인터페이스 이벤트의 상관관계를 검증한 경험은 Memory Controller나 Bus 검증과 충분히 맞닿아 있다고 생각합니다.

**꼬리 질문 대비 한 줄**
- `제가 주장하는 연결점은 HBM 지식 보유가 아니라 메모리 경로 검증 경험입니다.`

### 3. "UVM/SystemVerilog 기반 Testbench 구성이라고 했는데, 정확히 어디까지 하셨나요?"
**질문 의도**
- `구성`이란 단어의 실제 범위 확인
- 기존 환경 유지보수인지, 신규 설계인지, 소유 범위 확인

**답변 초안**
제 경험은 크게 두 가지입니다. 첫째, 기존 UVM 환경을 이해하고 필요한 Checker, Test, Report, 병렬화 구조를 확장하며 Block/TOP Regression을 운영한 경험입니다. 둘째, DataMover, Power Gating, VP 교차 검증처럼 특정 검증 목적에 필요한 In-house Checker 성격의 로직을 직접 추가한 경험입니다. 완전히 백지에서 범용 UVM 프레임워크를 혼자 새로 만든 경험만을 강조하기보다는, 실제 프로젝트에서 필요한 검증 기능을 환경에 붙이고 안정적으로 굴리는 데 강점이 있다고 말씀드리고 싶습니다.

**주의 포인트**
- `다 제가 다 만들었습니다` 식의 과장 금지
- 유지보수와 확장도 충분히 가치 있는 경험으로 설명

### 4. "In-house VIP 개발이라고 하셨는데, 상용 VIP 수준의 VIP를 직접 개발하신 건가요?"
**질문 의도**
- `VIP` 표현의 정확성 검증
- 용어를 부풀렸는지 확인

**답변 초안**
제가 말한 In-house VIP는 상용 벤더 VIP처럼 범용 패키지를 처음부터 만든 경험이라기보다, 우리 IP와 Spec에 맞춘 Checker와 검증 로직을 자체 구현해 활용한 경험에 가깝습니다. 예를 들어 DataMover 쪽에서는 Protocol Checker, Interrupt Checker, DM Checker를 운영했고, Power Gating 쪽에서는 상태 전이와 포트 값을 보는 Checker를 개발했습니다. 즉 범용 인터페이스 VIP 제품을 만든 것은 아니지만, 사내 검증 목적에 맞춘 전용 VIP 성격의 로직을 직접 설계하고 적용한 경험은 분명히 있습니다.

**꼬리 질문 대비 한 줄**
- `용어를 넓게 쓰기보다, 사내 Spec 맞춤형 검증 로직이라고 설명드리는 게 더 정확합니다.`

### 5. "Spec 기반 검증 계획 수립 경험을 구체적인 사례로 설명해 주세요."
**질문 의도**
- JD 핵심 역량 직접 검증
- Spec을 읽고 testcase로 내리는 사고력 확인

**답변 초안**
대표적인 사례는 DataMover와 Power Gating입니다. DataMover에서는 Base Address, Flow/Non-Flow 제어 방식, Interrupt 조건, 전송 간격 제약 같은 Spec 항목을 먼저 정리한 뒤, 정상 시나리오와 오류 시나리오를 분리해서 검증 포인트를 세웠습니다. Power Gating에서는 Normal, Retention, Power Down 상태 전이와 관련 포트 값을 기준으로 상태 전이 조건을 검증 항목으로 정의했습니다. 저는 Spec을 읽으면 먼저 observable한 신호와 이벤트로 쪼개고, 그다음 positive case, negative case, corner case, coverage 항목으로 내리는 방식으로 접근합니다.

**좋은 마무리 문장**
- `HBM에 들어가더라도 먼저 Spec을 검증 가능한 관찰 포인트로 분해하는 것부터 시작할 것입니다.`

### 6. "Coverage Closure를 수행했다고 했는데, 실제로 어떤 방식으로 닫아보셨나요?"
**질문 의도**
- `Closure`를 그냥 용어로 쓴 것인지, 실무적으로 이해하는지 확인
- Code coverage와 Functional coverage를 구분할 수 있는지 확인

**답변 초안**
제가 생각하는 Coverage Closure는 단순히 숫자를 올리는 작업이 아니라, 남아 있는 hole의 원인을 분석해서 sign-off 가능한 상태로 만드는 과정입니다. Code Coverage 쪽에서는 Verdi로 과제별 coverage를 확인하고, 왜 안 찍히는지 테스트 부족인지, 제약 문제인지, 환경 문제인지 먼저 구분합니다. Functional Coverage 쪽에서는 Register 조합을 사전에 분석하는 도구를 개발해서 미검증 조합을 찾고, 실제로 RTL bug도 한 건 발견해 보고했습니다. 실무적으로는 테스트 추가, 환경 수정, 필요 시 합리적인 제외 판단까지 포함해 닫는다는 관점으로 보고 있습니다.

**주의 포인트**
- 정확한 퍼센트를 모르면 말하지 말 것
- `waive`를 언급하더라도 본인이 실제 했는지 불확실하면 일반론 수준으로만 말할 것

### 7. "Regression 16시간을 2시간으로 줄였다고 했는데, 정확히 무엇을 바꾸신 건가요?"
**질문 의도**
- 수치 개선의 진위 확인
- 기술적인 설명 능력과 본인 기여도 확인

**답변 초안**
기존에는 Register Test가 직렬적으로 오래 도는 구조였는데, 저는 VCS 병렬성을 활용할 수 있도록 Job 단위를 더 잘게 나누고 Testbench 쪽 구조를 조정했습니다. 특히 Register File 단위로 분리해 병렬로 돌릴 수 있게 바꾸면서 병목을 줄였고, 그 결과 전체 Regression 시간이 약 16시간에서 2시간 수준으로 줄었습니다. 이 개선의 의미는 단순히 속도 향상보다, 하루에 돌려볼 수 있는 회수가 늘어나서 디버깅과 검증 사이클 전체가 빨라졌다는 점에 있습니다.

**꼬리 질문 대비 한 줄**
- `검증 효율 개선은 결국 더 많은 시도와 더 빠른 피드백으로 이어진다고 봤습니다.`

### 8. "QSLP/HWMPG Power Gating 검증 경험이 HBM 검증에 왜 의미가 있나요?"
**질문 의도**
- 서류의 다양한 경험 중 무엇이 직무와 연결되는지 확인
- Mixed/Top 관점 이해도 확인

**답변 초안**
Power Gating 검증의 핵심은 단순 기능 확인이 아니라 상태 전이의 안전성과 시스템 영향까지 보는 것입니다. 저는 SRAM 관련 포트와 상태 전이를 기준으로 Normal, Retention, Power Down 사이의 조건을 검증하는 Checker를 개발했고, HW-driven 제어 전환에 따라 새로운 시나리오를 반영했습니다. 이런 경험은 HBM에서도 단순 데이터 경로뿐 아니라 저전력, 상태 전이, Top 레벨 상호작용처럼 복합적인 검증 포인트를 이해하는 데 도움이 된다고 생각합니다. 특히 Mixed 또는 Top 관점에서 여러 조건이 얽힐 때 상태 기반으로 검증하는 사고방식이 강점입니다.

**주의 포인트**
- `HBM 저전력도 제가 바로 압니다` 식으로 확대 해석 금지

### 9. "D2D, PCIe, CXL, Co-emulation 경험은 없는 것 같은데 이 부분은 어떻게 보완할 건가요?"
**질문 의도**
- JD 미스매치 확인
- 부족한 부분을 대하는 태도와 학습 전략 확인

**답변 초안**
맞습니다. 그 영역은 직접 수행한 경험이 없습니다. 그래서 그 부분은 아는 척하기보다, 제가 이미 해온 검증 방식과 얼마나 빨리 연결할 수 있는지로 말씀드리는 것이 맞다고 생각합니다. 저는 새로운 도메인에 들어가면 먼저 Spec과 기존 Testbench 구조, 주요 시나리오, Coverage 정의를 빠르게 파악하고, 이후 로그와 Waveform 기반으로 실제 실패 패턴을 몸에 익히는 식으로 적응해왔습니다. 실제로 Modem TOP, VP, Power Gating, Register Coverage도 모두 같은 방식으로 빠르게 안착했기 때문에, 인터페이스 프로토콜 자체는 새로 배우더라도 검증 접근법은 빠르게 전개할 자신이 있습니다.

**좋은 마무리 문장**
- `부족한 도메인은 빠르게 메우고, 검증 방법론은 바로 기여하는 형태가 제 현실적인 포지션입니다.`

### 10. "왜 지금 SK하이닉스로 옮기려고 하나요? 삼성전자에서 계속해도 되지 않나요?"
**질문 의도**
- 이직 사유의 건강성 확인
- 조직 적응성과 장기 근속 가능성 확인

**답변 초안**
이직 이유를 부정적으로 설명하고 싶지는 않습니다. 삼성전자에서 8년 동안 Block/TOP 검증, DMA, Power Gating, Coverage, 자동화까지 정말 많은 것을 배웠고 그 경험은 제 기반이 되었습니다. 다만 제가 쌓아온 검증 역량을 메모리 중심의 더 큰 스케일에서 직접 활용해 보고 싶다는 생각이 커졌고, 그 지점에서 HBM Digital Design 검증은 제 경험과 가장 잘 연결되는 기회라고 판단했습니다. 즉 회사를 떠나고 싶다기보다, 지금까지 쌓은 검증 방법론을 가장 임팩트 있게 쓸 수 있는 무대로 가고 싶다는 것이 핵심입니다.

**주의 포인트**
- 조직 불만, 보상 불만, 타 부서 갈등 언급 금지
- `삼성도 좋았지만 이제 더 맞는 문제를 풀고 싶다`로 정리

## 추가 메모
### 면접에서 먼저 깔고 가면 좋은 포지션
- `HBM 자체 경력자`가 아니라 `HBM에 빠르게 전개 가능한 강한 디지털 검증 엔지니어`
- `범용 툴 사용자`보다 `Spec을 검증 항목으로 내리고 자동화와 체커로 연결하는 사람`
- `말을 크게 하는 사람`보다 `한 경험을 정확히 설명하는 사람`

### 대답 길이 가이드
- 첫 답변: 30초 내 핵심 결론
- 꼬리 질문: 30~60초 사례 확장
- 너무 길어지면 마지막에 `정리하면 ...`으로 한 번 묶기

### 절대 피해야 할 표현
- `다 해봤습니다`
- `HBM도 사실 비슷합니다`
- `전 회사에서 한계가 있어서 나옵니다`
- `VIP를 처음부터 다 만들었습니다` (불확실할 경우)
