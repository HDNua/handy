# Functional coverage
DataMover 검증에서 Function coverage 란, DM Spec 에 기술된 모든 client 들이 한 번 이상 test 되었는지를 판정하는 것입니다.
.
아주 다양한 client 가 있으며, 개중에는 N/A 인 클라이언트도 있습니다.
하나 주목할 점은, 가장 짧은 길이의 interval 은 110 정도 된다는 것이고, 가장 긴 interval 도 1000000 을 넘지 않는다는 것입니다.
사실, 이는 한 전송 간격의 최대 길이가 1ms 이므로 당연한 것입니다. (1 TTI = 1 ms)
.
.
.
# Error 분석
.
…(1) INTERVAL_ERROR
…> Interrupt 간의 간격이 20 us 이내인 경우
…(2) DATA_UNMATCHED
…> Data가 맞지 않는 경우
…>> (2.1) 시뮬레이션 극 후반에 나서 Memory에 쓰여지기 전 Simulation 이 종료된 경우
…>> (2.2) Flow control client 의 경우 interrupt error 에 대한 제약이 없는데, 너무 가까이 뜨는 경우 script 에서 채지 못하고 다음 int_cnt 의 data 와 비교하는 경우가 있습니다. 이는 고쳐야 할 사항으로, 주로 SCH 에서 비슷한 경우가 발생합니다.
…>> (2.3) 실제로 bug 여서 RTL 수정이 필요한 경우
…(3) NO_TOUCH_KEY
…> Touch 되지 않은 key 가 있는 경우
…>> 현재 이 에러는 거의 발생하지 않습니다.
…(4) NO_TOUCH_FILE
…> Touch 되지 않은 file 이 있는 경우
…>> (4.1) DM spec 에서 File 이름이 잘못된 경우
…>> (4.2) DM spec 에서 Interrupt 또는 Offset 이 잘못 기술되어 Memory 에 쓰여졌으나 체크를 못한 경우
…(5) NO_DATA_IN_MEM
…> Client 에서 쏜 데이터가 Memory에 쓰이는 transaction 으로서 보이지 않음
…(6) NO_INTERRUPT
…> Client 에서 transaction 이 발생했으나 이와 matching 되는 interrupt 가 뜨지 않은 경우
…>> (6.1) DM spec 에서 File 이름이 잘못된 경우
…>> (6.2) DM spec 에서 Interrupt 또는 Offset 이 잘못 기술되어 Memory 에 쓰여졌으나 체크를 못한 경우
…(7) NO_BURST
…> Interrupt 가 떴으나 이에 맞는 transaction 이 없는 경우
…>> (7.1) Interrupt 가 떴으나 이에 맞는 Data transaction 이 run 폴더 내의 file 들에서 찾을 수 없는 경우
