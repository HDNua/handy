15. `Regression 16h -> 2h` 단축은 기술적으로 어떻게 달성했나요?
(15.1) 재료 정리 및 생각
(260328-2125) SFR Test, Special Function Register 에 대한 테스트 최적화 관련이다.
기본적으로는 UVM register model 에 존재하는 모든 Address Map 의, 모든 Register File 의, 모든 Register 를 순회하면서
reset value 일치성을 보거나, random value 를 WR, 이후 bit-wise negate 하여 다시 WR 하여 정합성을 검증한다.
기존 환경은 Cadence 의 Xcelium 기반으로 구성되어 regression 을 돌리기 위한 Job 수가 제한적이었고,
이로 인해 대다수의 Register 가 기준 없이 임의의 덩어리로 나뉜 채 serial 로 주루룩 검증되어 검증 시간이 매우 길고 report 파악이 어려웠다.
내가 SFR test 업무를 받고 1년 정도 후, 작업 환경을 Xcelium 에서 Synopsys 의 VCS Tool 로 옮기게 되었는데,
VCS 를 사용하는 경우 Xcelium 대비하여 regression 에 사용 가능한 Job 수가 크게 늘었다는 것을 알게 되었다.
Modem Register 검증은 기본적으로 대부분의 레지스터가 독립이며 단순 WR 확인이었기 때문에,
Register File 단위로 분리하여 regression 을 늘리면 Job 수를 더 많이 사용하는 대신 regression time 이 크게 줄 것으로 예상했다.
이 생각은 적중했으며, 덤으로 reporting 까지 크게 강화되어 이후 SFR test 는 거의 완전한 enter job 이 되었다.
16h 는 걸리던 sim이 2h 로 줄었기 때문에, 하루에 한 번 가능하던 test 를 여러 번 돌릴 수 있게 된 것 또한 큰 장점.
